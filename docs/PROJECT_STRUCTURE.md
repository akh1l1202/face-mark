# Project Structure

**Project:** Face-Mark

**Version:** 1.0

---

# 1. Overview

Face-Mark follows a **monorepo architecture**, where both the Flutter frontend and FastAPI backend are maintained within a single repository. This structure simplifies version management, keeps documentation centralized, and allows frontend and backend changes to evolve together.

The repository is organized into three primary areas:

* **Frontend** тАУ Flutter mobile application
* **Backend** тАУ FastAPI server and facial recognition engine
* **Documentation** тАУ Project specifications and technical documentation

---

# 6. `main.dart`

## Purpose

Application entry point.

Responsibilities:

* Initialize Flutter
* Configure Firebase
* Register application services
* Define application routes
* Launch the root widget

Every application execution begins here.

---

# 7. `api_service.dart`

## Purpose

Acts as the communication layer between Flutter and the FastAPI backend.

Responsibilities:

* Send HTTP requests
* Receive API responses
* Serialize and deserialize JSON
* Handle API errors
* Manage request timeouts

This file should contain **all backend communication logic**, keeping UI code independent of networking.

---

# 8. `attendance_service.dart`

## Purpose

Manages attendance-related business logic on the client.

Responsibilities:

* Synchronize attendance data
* Coordinate with Firestore
* Trigger attendance updates
* Process backend responses
* Maintain attendance state

Separating attendance logic from the user interface improves maintainability and testability.

---

# 15. `main.py`

Primary FastAPI application.

* API routing
* Request validation
* Face recognition
* Attendance creation
* Teacher registration
* Response generation

---

# 16. `firebase_utils.py`

Contains Firebase Admin SDK functionality.

Responsibilities:

* Firebase initialization
* Firestore access
* Push notification delivery
* Helper methods for Firebase operations

Keeping Firebase functionality isolated improves modularity.

---

# 17. `requirements.txt`

Lists Python dependencies required by the backend.

Typical dependencies include:

* FastAPI
* Uvicorn
* NumPy
* OpenCV
* face_recognition
* Firebase Admin SDK

---

# 18. `embeddings.json`

Stores facial embeddings generated during teacher enrollment.

Each record contains:

* Teacher identifier
* Facial embedding vector
* Associated metadata

Current implementation uses local JSON storage for simplicity.

This file is ignored by Git because it contains generated runtime data.

---

# 19. `profile_photos/`

Stores cropped profile images generated during enrollment.

Uses:

* Recognition reference
* Administrator display
* Debugging

Like `embeddings.json`, this directory is excluded from version control.

# 22. Development Conventions

## Frontend

* UI components belong in `widgets/`
* Screens belong in `screens/`
* Networking belongs in `api_service.dart`
* Attendance logic belongs in `attendance_service.dart`

Business logic should remain separate from presentation code.

---

## Backend

* Route handling remains in `main.py`
* Firebase operations remain in `firebase_utils.py`
* Runtime data should never be committed to Git

---

## Documentation

Every major architectural or feature change should be reflected in the corresponding document within the `docs/` directory.

Documentation should evolve alongside the codebase.
