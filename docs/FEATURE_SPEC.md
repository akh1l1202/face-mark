**Project:** Face-Mark

**Version:** 1.0

# Introduction

This document defines every functional feature implemented in Face-Mark. Each feature includes its purpose, user workflow, backend responsibilities, validation rules, edge cases, and acceptance criteria.

The goal is to provide sufficient detail for developers to implement or maintain the system consistently.

# Feature 1 вҖ” Administrator Authentication

## Purpose

Provide secure access to administrative functionality. Only authenticated administrators may register teachers, manage attendance, or configure the system.

---

## Actors

* Administrator

---

## Preconditions

* Administrator account exists.
* Internet connection is available.
* Firebase Authentication is configured.

---

## Workflow

1. Administrator launches the application.
2. Login screen is displayed.
3. Administrator enters credentials.
4. Credentials are validated.
5. Firebase Authentication verifies identity.
6. On success, administrator is redirected to the dashboard.
7. On failure, an error message is displayed.

---

## Functional Requirements

The system shall:

* Allow administrator login.
* Reject invalid credentials.
* Maintain authenticated sessions.
* Support logout.
* Redirect unauthenticated users to the login screen.

---

## Validation

* Email must be valid.
* Password cannot be empty.
* Firebase authentication errors must be handled gracefully.

---

## Error States

| Error                | Expected Behaviour                         |
| -------------------- | ------------------------------------------ |
| Invalid password     | Display authentication error               |
| Unknown account      | Prompt administrator to verify credentials |
| Network unavailable  | Inform user and allow retry                |
| Firebase unavailable | Display service unavailable message        |

---

## Acceptance Criteria

* Administrator can log in successfully.
* Invalid credentials are rejected.
* Session persists until logout.
* Unauthorized users cannot access protected screens.

# Feature 3 вҖ” Face Enrollment

## Purpose

Capture and register facial data for each teacher to enable automatic recognition during attendance.

---

## Actors

* Administrator

---

## Workflow

1. Administrator selects a teacher.
2. Opens the face registration screen.
3. Camera preview starts.
4. System detects the teacher's face.
5. Multiple images may be captured to improve recognition quality.
6. Facial embeddings are generated by the backend.
7. Embeddings are stored.
8. Enrollment is marked as complete.

---

## Functional Requirements

The system shall:

* Detect a face before capture.
* Reject images with no detectable face.
* Reject poor-quality images.
* Generate facial embeddings.
* Associate embeddings with the selected teacher.
* Allow re-enrollment if needed.

---

## Validation Rules

* Exactly one face should be visible during enrollment.
* Image quality must meet minimum standards.
* Enrollment cannot proceed without successful embedding generation.

---

## Edge Cases

* No face detected.
* Multiple faces detected.
* Camera unavailable.
* Backend unavailable.
* Embedding generation failure.
* Duplicate enrollment request.

---

## Acceptance Criteria

* Valid face images are enrolled successfully.
* Embeddings are stored correctly.
* Invalid captures are rejected with clear feedback.
* Administrators can retry enrollment after failure.

# Feature 4 вҖ” Face Recognition

## Purpose

Automatically identify registered teachers by comparing a live camera capture with previously enrolled facial embeddings.

This feature is the core intelligence of Face-Mark and is responsible for replacing traditional attendance methods with a touchless, AI-powered identification process.

---

## Actors

* Teacher
* Face Recognition Engine
* FastAPI Backend
* Firebase Firestore

---

## Components Involved

### Frontend

* Camera Preview
* `api_service.dart`
* `attendance_service.dart`

### Backend

* `main.py`
* `embeddings.json`
* `profile_photos/`

### Libraries

* Google ML Kit (Face Detection)
* face_recognition
* NumPy
* OpenCV

---

# Recognition Workflow

## Step 1 вҖ” Camera Detection

The kiosk continuously monitors the camera feed.

Google ML Kit detects faces locally on the device.

If no face exists:

* Continue scanning.

If a face exists:

* Capture the frame.
* Send image to FastAPI.

---

## Step 2 вҖ” Backend Processing

FastAPI receives the image.

The backend:

1. Detects face location.
2. Crops the face.
3. Generates a facial embedding.
4. Normalizes the embedding.
5. Loads all registered teacher embeddings.
6. Computes similarity scores.
7. Finds the closest match.

---

## Step 3 вҖ” Decision

### Match Found

If similarity exceeds the configured threshold:

* Teacher identified.
* Attendance workflow begins.

### No Match

If similarity is below threshold:

* Return "Unknown".
* Attendance is not recorded.

---

# Functional Requirements

The system shall:

* Detect faces in real time.
* Generate facial embeddings.
* Compare against all registered teachers.
* Identify the closest valid match.
* Reject unknown faces.
* Process recognition within acceptable time limits.

---

# Matching Logic

The backend compares the generated embedding with every registered teacher embedding.

For each comparison:

* Compute distance.
* Calculate similarity.
* Select smallest distance.
* Verify confidence threshold.

If confidence passes the threshold:

Return teacher ID.

Otherwise:

Return Unknown.

---

# Validation Rules

The system must reject recognition when:

* No face detected.
* Multiple faces detected.
* Face partially visible.
* Image blurred.
* Backend processing fails.
* Recognition confidence below threshold.

---

# Performance Targets

| Metric               | Target     |
| -------------------- | ---------- |
| Face detection       | <300 ms    |
| Backend processing   | <700 ms    |
| Total recognition    | <2 seconds |
| Recognition accuracy | >95%       |

---

# Edge Cases

### No Face

Continue scanning.

---

### Multiple Faces

Reject recognition.

Display:

> Multiple faces detected.

---

### Teacher Wearing Glasses

Recognition should still succeed if enrolled under similar conditions.

---

### Poor Lighting

Reject frame.

Continue scanning until image quality improves.

---

### Unknown Person

Display:

> Teacher not recognized.

Do not mark attendance.

---

### Camera Failure

Stop recognition.

Notify administrator.

---

# Error Handling

| Condition           | Action               |
| ------------------- | -------------------- |
| Camera unavailable  | Retry initialization |
| Backend timeout     | Retry request        |
| Invalid response    | Ignore frame         |
| Embedding failure   | Log error            |
| Recognition failure | Continue scanning    |

---

# Acceptance Criteria

The feature is complete when:

* Registered teachers are identified accurately.
* Unknown users are rejected.
* Multiple faces are handled correctly.
* Recognition occurs within performance targets.
* Attendance is triggered only after successful identification.

# Feature 5 вҖ” Attendance System

## Purpose

Automatically record teacher attendance immediately after successful facial recognition.

Attendance should require no manual interaction from the teacher.

---

## Actors

* Teacher
* FastAPI Backend
* Firestore
* Administrator

---

# Workflow

## Entry

1. Teacher is recognized.
2. Backend verifies today's attendance.
3. If no active session exists:

   * Create attendance record.
   * Mark Entry Time.
   * Notify administrator.

---

## Exit

1. Teacher is recognized again.
2. Backend detects an active attendance session.
3. Exit Time is recorded.
4. Attendance session closes.

---

# Attendance Record

Each attendance record is synchronized to Firestore and contains the following fields *(for structural definitions, refer to the [Database Schema Specification](file:///D:/GitHub/face-mark/docs/DATABASE_SCHEMA.md#5-attendance-collection)):*

* Teacher ID
* Teacher Name
* Date
* Entry Time
* Exit Time
* Attendance Status
* Recognition Confidence
* Timestamp

---

# Functional Requirements

The system shall:

* Record entry time.
* Record exit time.
* Prevent duplicate entries.
* Maintain attendance history.
* Synchronize records with Firestore.
* Notify administrators.

---

# Attendance States

## Not Present

Teacher has not been recognized today.

---

## Checked In

Entry time exists.

Exit time not yet recorded.

---

## Checked Out

Both entry and exit recorded.

Attendance session complete.

---

---

# Duplicate Prevention

The backend shall:

* Check existing attendance for the current day.
* Prevent multiple entry records.
* Prevent multiple exit records.
* Ignore repeated recognition within a configurable cooldown period.

---

# Validation Rules

Attendance cannot be created when:

* Teacher is unknown.
* Recognition confidence below threshold.
* Backend validation fails.
* Duplicate entry detected.

---

# Notifications

On successful attendance:

Administrator receives:

* Teacher Name
* Attendance Type
* Timestamp

---

# Edge Cases

### Teacher recognized repeatedly

Ignore repeated detections during cooldown.

---

### Backend unavailable

Queue retry if possible.

Otherwise notify administrator.

---

### Firestore unavailable

Do not lose attendance event.

Retry synchronization.

---

### Recognition immediately after enrollment

Should succeed without requiring application restart.

---

# Performance Targets

| Metric                | Target     |
| --------------------- | ---------- |
| Attendance creation   | <500 ms    |
| Firestore sync        | <1 second  |
| Notification delivery | <3 seconds |

---

# Acceptance Criteria

The feature is complete when:

* Entry attendance is recorded correctly.
* Exit attendance is recorded correctly.
* Duplicate attendance is prevented.
* Attendance synchronizes with Firestore.
* Notifications are delivered successfully.

# Feature 11 вҖ” Error Handling

## Purpose

Ensure Face-Mark remains stable, predictable, and user-friendly when unexpected situations occur. The application should recover gracefully whenever possible and provide clear feedback to administrators without exposing internal implementation details.

---

# Error Handling Principles

The system shall:

* Never crash due to recoverable errors.
* Display clear and actionable error messages.
* Log unexpected exceptions for debugging.
* Retry transient failures where appropriate.
* Prevent data corruption or duplicate attendance records.

---

## Frontend Errors

### Camera Initialization Failure

**Possible Causes**

* Camera permission denied.
* Camera hardware unavailable.
* Camera already in use.

**Expected Behaviour**

* Display an informative message.
* Provide a retry option.
* Prevent face recognition until the camera is available.

---

### Internet Connectivity Loss

**Possible Causes**

* Device offline.
* Weak network connection.

**Expected Behaviour**

* Inform the administrator.
* Retry requests automatically when connectivity is restored.
* Prevent duplicate API requests.

---

### Authentication Failure

**Possible Causes**

* Invalid credentials.
* Expired session.
* Firebase Authentication unavailable.

**Expected Behaviour**

* Redirect to the login screen if the session expires.
* Allow the administrator to retry authentication.

---

## Backend Errors

### Invalid API Request

Examples:

* Missing required fields.
* Invalid JSON.
* Incorrect request format.

**Response**

* Return an appropriate HTTP status code.
* Include a descriptive error message.
* Reject the request without affecting existing data.

---

### Face Recognition Failure

Possible causes:

* No detectable face.
* Multiple faces detected.
* Face encoding generation failed.
* Recognition confidence below threshold.

**Expected Behaviour**

* Reject the recognition attempt.
* Do not create an attendance record.
* Continue monitoring for the next valid frame.

---

### Firestore Synchronization Failure

Possible causes:

* Network interruption.
* Firebase service outage.
* Permission errors.

**Expected Behaviour**

* Retry synchronization where appropriate.
* Preserve the attendance event until synchronization succeeds.
* Notify the administrator if manual intervention is required.

---

### Notification Failure

Possible causes:

* Invalid FCM token.
* Firebase Cloud Messaging unavailable.

**Expected Behaviour**

* Attendance processing should continue even if notifications fail.
* Log the failure for later investigation.

---

## Error Messages

The application should use simple, user-friendly messages.

| Condition               | Message                                                                 |
| ----------------------- | ----------------------------------------------------------------------- |
| Camera unavailable      | Camera could not be started. Please try again.                          |
| No face detected        | No face detected. Please stand in front of the camera.                  |

---

## Logging

The system should log:

* Authentication failures.
* Recognition failures.
* Attendance creation events.
* Backend exceptions.
* Firestore synchronization errors.
* Notification failures.

Logs should assist developers in diagnosing issues without exposing sensitive information to end users.

# Feature 12 вҖ” Edge Cases

## Purpose

Define system behavior for uncommon but expected scenarios to ensure consistent operation.

---

## Face Recognition Edge Cases

### No Face Visible

**Scenario**

The camera detects no face.

**Expected Behaviour**

* Continue scanning.
* Do not send unnecessary backend requests.

---

### Multiple Faces

**Scenario**

More than one face is visible.

**Expected Behaviour**

* Reject recognition.
* Inform the user.
* Resume scanning once only one face remains.

---

### Face Partially Visible

**Scenario**

Only part of the face is visible.

**Expected Behaviour**

* Wait until the full face is visible.
* Do not attempt recognition.

---

### Poor Lighting

**Scenario**

Lighting conditions prevent reliable recognition.

**Expected Behaviour**

* Request the user to improve lighting or reposition.
* Continue monitoring until image quality improves.

---

### Teacher Wearing Accessories

**Scenario**

Teacher is wearing glasses, a mask, or other accessories.

**Expected Behaviour**

* Attempt recognition using available facial features.
* If confidence is insufficient, reject the attempt.

---

## Attendance Edge Cases

### Duplicate Recognition

**Scenario**

The same teacher remains in front of the kiosk after attendance is recorded.

**Expected Behaviour**

* Ignore repeated detections during the configured cooldown period.
* Do not create duplicate attendance records.

---

### Immediate Re-entry

**Scenario**

Teacher briefly leaves and returns.

**Expected Behaviour**

* Respect cooldown logic.
* Prevent accidental duplicate check-ins.

---

### Attendance Without Registration

**Scenario**

An unregistered person stands in front of the kiosk.

**Expected Behaviour**

* Do not create attendance.
* Display an "Unknown Teacher" message.
* Optionally log the event for review.

---

## Administrator Edge Cases

### Delete Teacher with Existing Attendance

**Expected Behaviour**

* Preserve historical attendance records.
* Remove only the active teacher profile and associated face data.

---

### Edit Teacher During Active Attendance

**Expected Behaviour**

* Allow profile updates that do not affect attendance integrity.
* Prevent inconsistent data states.

---

## Backend Edge Cases

### Backend Restart During Recognition

**Expected Behaviour**

* Reject in-progress requests gracefully.
* Resume processing automatically after the service becomes available.

---

### Corrupted Embeddings

**Expected Behaviour**

* Skip invalid embeddings.
* Log the issue.
* Continue recognition using remaining valid data.

---

### Missing Profile Photo

**Expected Behaviour**

* Continue recognition.
* Display a default avatar where required.

---

## Security Edge Cases

### Unauthorized API Access

**Expected Behaviour**

* Reject the request.
* Return an appropriate HTTP status code.
* Log the attempt.

---

### Invalid Firebase Token

**Expected Behaviour**

* Reject authenticated operations.
* Require the administrator to authenticate again.

---

## Acceptance Criteria

The system satisfies this specification when:

* Recoverable errors do not crash the application.
* Attendance integrity is preserved under all supported scenarios.
* Clear feedback is provided to administrators.
* Recognition failures never create invalid attendance records.
* Logs are generated for diagnostic purposes.
* All defined edge cases produce predictable and documented behavior.
